## 1. Architecture Overview

The plugin consists of three layers:

**Layer 1 — MCP Server** (`servers/github-memory-server.js`): the single point of contact with the GitHub API. Provides Claude with a set of MCP tools for reading, writing, and searching in the repository. Maintains session state (author cache, current project, SHA).

**Layer 2 — Skill** (`skills/shared-memory/SKILL.md`): teaches Claude when and how to use MCP tools. Contains matching algorithms, UX patterns, and rules for separation from local memory. This is the "brain" of the plugin — all logic requiring LLM reasoning.

**Layer 3 — Commands** (`commands/*.md`): slash commands for direct access (`/memory`, `/remember`, `/project`). Thin wrappers over MCP tools.

### Principle of Separation of Responsibility

| What it handles | MCP Server | Skill | Commands |
|---|---|---|---|
| Parsing root.md | ✅ | — | — |
| Atomic commits | ✅ | — | — |
| SHA verification, retry | ✅ | — | — |
| Slugify, escaping | ✅ | — | — |
| Author cache | ✅ | — | — |
| Auto-discover Related links | ✅ | — | — |
| Throttling parallel requests | ✅ | — | — |
| Keyword extraction | — | ✅ | — |
| Intent determination (search/write/update) | — | ✅ | — |
| Project vs _shared selection | — | ✅ | — |
| Separation from local memory | — | ✅ | — |
| UX formulations (confirmations, warnings) | — | ✅ | — |
| Quick call to specific function | — | — | ✅ |

## 3. MCP Server: Detailed Design

### 3.1 Configuration (.mcp.json)

```json
{
  "mcpServers": {
    "shared-memory": {
      "type": "stdio",
      "command": "node",
      "args": ["${CLAUDE_PLUGIN_ROOT}/servers/github-memory-server.js"],
      "env": {
        "GITHUB_TOKEN": "{{GITHUB_TOKEN}}",
        "GITHUB_REPO": "{{GITHUB_REPO}}"
      }
    }
  }
}
```

The variables `GITHUB_TOKEN` and `GITHUB_REPO` are plugin settings, set by the user during installation. The format of `GITHUB_REPO` is: `owner/repo-name`.

> **Token storage:** Cowork stores plugin settings securely (not in plaintext files). On first plugin load, user enters token in the plugin settings dialog. Token is validated by `connect_repo` on first call (attempts `GET /user`). If invalid, error is returned with guidance to update settings.
>
> **Token scopes:** `repo` scope gives read/write access to private and public repositories. For team use, consider creating a shared bot account with limited permissions.
>
> **Settings path in Cowork:** Settings → Installed Plugins → Shared Memory → Configure. No restart needed after updating token — next `connect_repo` call uses the new value.

### 3.3 MCP Server Initialization

```javascript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "shared-memory",
  version: "1.0.0"
});

// Example 1 — tool without parameters (connect_repo):
server.registerTool("connect_repo", {
  title: "Connect Repository",
  description: "Connect to the shared memory GitHub repository",
  inputSchema: z.object({}),  // empty Zod object, NOT just {}
  annotations: { readOnlyHint: false, idempotentHint: true }
}, async () => {
  const result = await connectToRepo();
  return {
    content: [{ type: "text", text: JSON.stringify(result) }],
    structuredContent: result  // structured output for machine-readable access
  };
});

// Example 2 — tool with parameters (read_root):
server.registerTool("read_root", {
  title: "Read Root Index",
  description: "Read root.md index for a project",
  inputSchema: z.object({
    project: z.string().describe("Project folder name: '_shared', 'mobile-app', etc.")
  }),
  annotations: { readOnlyHint: true, idempotentHint: true }
}, async ({ project }) => {
  const result = await readRootMd(project);
  return {
    content: [{ type: "text", text: JSON.stringify(result) }],
    structuredContent: result
  };
});

// Example 3 — tool with optional parameters (write_entry):
server.registerTool("write_entry", {
  title: "Write Entry",
  description: "Create a new entry in shared memory",
  inputSchema: z.object({
    project: z.string(),
    title: z.string(),
    content: z.string(),
    tags: z.array(z.string()),
    description: z.string().max(80),
    auto_related: z.boolean().optional().default(true),
    related_override: z.array(z.string()).optional()
  }),
  annotations: { readOnlyHint: false, destructiveHint: false }
}, async (params) => {
  const result = await createEntry(params);
  return {
    content: [{ type: "text", text: JSON.stringify(result) }],
    structuredContent: result
  };
});

const transport = new StdioServerTransport();
await server.connect(transport);
```

> **API Reference (MCP SDK v1.27+):**
> - `zod` — peer dependency of SDK (already included in package.json)
> - Each tool is registered via `server.registerTool(name, definition, handler)`
> - `inputSchema` **must be a Zod object** (`z.object({...})`), not a plain JS object
> - Handler returns `{ content: [{ type: "text", text }], structuredContent?: object }`
> - `structuredContent` — optional modern pattern for machine-readable output
>
> **Convention:** Output specifications in section 3.4 below show **structuredContent** (the JSON structure Claude sees). The handler wraps it in `JSON.stringify()` + `content: [{ type: "text", text }]`.
>
> **Tool naming:** Tools are registered as `read_root`, `write_entry`, etc. Claude sees them with the prefix `mcp__shared-memory__read_root`, `mcp__shared-memory__write_entry` — the prefix is added automatically by the server name from `.mcp.json`.

### 3.4 Standard Error Output Format

All tools return errors in a consistent format. When a tool encounters an error, the handler returns:

```javascript
{
  content: [{ type: "text", text: JSON.stringify(errorResult) }],
  structuredContent: errorResult,
  isError: true  // signals to Claude that the tool call failed
}
```

Where `errorResult` follows this shape:

```
{
  status: "error",
  error_code: string,       // machine-readable codes:
                             //   "auth_failed", "repo_not_found", "rate_limit_rest" (5000/hr),
                             //   "rate_limit_search" (10/min), "network_error", "sha_conflict",
                             //   "parse_error", "not_found", "concurrent_edit"
  error: string,             // human-readable message (English)
  retry_possible: boolean,   // hint to skill: can this be retried?
  retry_after_ms?: number    // if retry_possible, suggested wait time
}
```

The skill uses `error_code` to select the appropriate bilingual UX message from `references/error-handling.md`.

### 3.5 MCP Tools — Complete List

> **Note on section references:** Tools reference spec functions (F1–F9) from the
> [Functional Requirements](shared-memory-plugin-requirements.md).

#### Tool: `connect_repo`
**Spec compliance:** F1
**When called:** at session start (automatically via skill) or on user command.

```
Input:  {} (no parameters — token and repo from env)
Output: {
  status: "connected" | "initialized" | "error",
  user: { name: string, login: string },
  projects: string[],           // list of projects (excluding _shared, _meta)
  shared_entries_count: number,  // number of entries in _shared/root.md
  error?: string
}
```

Logic:
1. `GET /user` → determine author name (cache for session)
2. `GET /repos/{owner}/{repo}/contents/` → verify presence of `_meta.md` and `_shared/root.md`
3. If empty → cold start: create `_meta.md` + `_shared/root.md` in a single atomic commit → return `status: "initialized"`
4. If partially initialized → create missing files → return `status: "initialized"`
5. Read `_shared/root.md`, count entries → return `status: "connected"`
6. Return list of projects

**Idempotency:** calling `connect_repo` multiple times in a session is safe. After first call, subsequent calls skip initialization and return cached author + fresh project list.

#### Tool: `read_root`
**Spec compliance:** F2
**When called:** on every memory access (skill calls it fresh each time).

```
Input:  { project: string }   // folder name: "_shared", "mobile-app", etc.
Output: {
  project: string,
  description: string,        // project description (first line of root.md)
  entries: [
    { name: string, file: string, description: string, tags: string[] }
  ],
  raw_markdown: string         // for cases where Claude needs full text
}
```

Logic:
1. `GET /repos/{owner}/{repo}/contents/{project}/root.md`
2. Decode base64 → parse markdown table → return structure
3. On parse error → fallback mode:
   - List all .md files in directory (excluding root.md) via getDirectoryListing()
   - Return `{ entries: [{ file: "name.md" }], corrupted: true, raw_markdown: "<original>" }`
   - Skill detects `corrupted: true` → warns user, disables auto_related, disables write_entry for this project (root.md must be fixed first)

#### Tool: `read_entry`
**Spec compliance:** F2 (loading entries)
**When called:** when Claude identifies a relevant entry and wants to load its content.

```
Input:  { project: string, file: string }  // "mobile-app", "rive-vs-lottie.md"
Output: {
  title: string,
  date: string,
  author: string,
  tags: string[],
  content: string,     // full entry content (markdown)
  sha: string,         // file SHA (for subsequent update)
  related: string[]    // structured list of Related links (e.g., ["stack.md", "../_shared/deploy-process.md"])
}
```

#### Tool: `write_entry`
**Spec compliance:** F3
**When called:** after Claude prepares an entry and receives user confirmation.

```
Input: {
  project: string,           // "mobile-app" or "_shared"
  title: string,             // entry title
  content: string,           // content (markdown, without frontmatter)
  tags: string[],            // tags
  description: string,       // brief description for root.md (max 80 chars)
  auto_related: boolean,     // true = tool finds Related automatically (default true)
  related_override?: string[] // manual override of Related list (ignores auto_related)
}
Output: {
  status: "created" | "conflict" | "error",
  file: string,              // created file (slug)
  commit_sha: string,
  related_added: string[],   // which Related links were added automatically
  related_candidates?: [     // if auto_related=true and >3 candidates — skill shows for selection
    { file: string, common_tags: string[] }
  ],
  retry_count?: number,      // how many retries were needed
  duplicate_candidates?: [   // if similar entries found
    { file: string, common_tags: string[], keyword_overlap: number }
  ],
  error?: string
}
```

Logic (atomic commit):
1. Read current HEAD SHA
2. Read current root.md
3. Generate slug from title → check uniqueness → add suffix if needed
4. Collect entry content: frontmatter (Date, Author, Tags) + content + Related
5. Auto-discover Related: if auto_related=true → read project root.md + _shared, find entries with ≥1 common tag. If 1-3 → add links automatically (section `## Related`). If >3 → return candidates in output (skill shows to user for selection). If related_override → use it instead of auto-discovery
6. Form new row for root.md (with escaping of `|`)
7. Create blobs (new file + updated root.md)
8. Create tree with both files (**must include `base_tree`** parameter = SHA of current tree from parent commit, to preserve existing files in repository)
9. Create commit (parent = HEAD SHA)
10. Update ref (refs/heads/main)
11. If SHA mismatch on step 10 → retry from step 1 (up to 3 times). On retry `addEntryToRoot()` checks that entry has not yet been added (idempotency — returns `was_added: false` if row already exists, meaning the previous attempt partially succeeded)

**Related override flow:** When skill calls write_entry a second time with `related_override` (after user selects from candidates), this is a FULL new atomic commit. The idempotency check in addEntryToRoot prevents duplicate rows. The entry file is re-created with the correct Related section. This is safe because the entry was already committed in the first call — the second call updates only the Related section via a new commit.

#### Tool: `update_entry`
**Spec compliance:** F5

```
Input: {
  project: string,
  file: string,             // entry filename
  previous_sha: string,     // SHA from read_entry (for conflict detection)
  new_content: string,      // updated content
  new_tags?: string[],      // updated tags (optional)
  new_description?: string  // updated description for root.md (optional)
}
Output: {
  status: "updated" | "conflict" | "concurrent_edit" | "error",
  commit_sha: string,
  current_sha?: string,      // current file SHA (on concurrent_edit, for retry)
  previous_author?: string,  // if entry was changed by someone else
  previous_date?: string,
  diff_summary?: string,     // brief summary of what changed since previous_sha
  error?: string
}
```

Logic:
1. Read current file version from GitHub (always, ignoring context)
2. Compare current file SHA with `previous_sha` from input:
   - If SHAs match → safe to proceed
   - If SHAs differ → return `concurrent_edit` with `current_sha`, `previous_author`, `previous_date`, and `diff_summary` (skill shows this to user and asks whether to proceed)
3. Prepare updated file (frontmatter + new_content)
4. If new_tags/new_description → update corresponding row in root.md
5. If new_tags changed → recalculate Related links (auto-discover again with new tags), update `## Related` section in file.
   **Note:** Related link updates modify only the entry being updated, not the entries it links TO. Cross-entry link consistency is not atomically guaranteed. If related entries are updated concurrently, links may become one-directional until next update. This is acceptable — cyclic links are non-critical (see Open Question #16).
6. Atomic commit (similar to write_entry)

#### Tool: `search_tags`
**Spec compliance:** F4 (fast search)

```
Input: {
  keywords: string[],         // keywords (Claude extracts from query)
  active_project?: string     // for result prioritization
}
Output: {
  results: [
    {
      project: string,
      file: string,
      name: string,
      description: string,
      tags: string[],
      match_count: number,     // number of matching keywords
      match_details: string[]  // which words matched
    }
  ],
  total_count: number,          // actual total matches (may exceed results.length)
  was_truncated: boolean,       // true if total_count > 15 (results capped at 15)
  searched_projects: string[]   // which projects were checked (includes archived)
}
```

Logic:
1. Read root.md of all projects + _shared + archived projects (in parallel via Promise.all)
2. For each entry: compare keywords with Tags (exact) and Description (substring)
3. Rank: match_count DESC, then priority (active > _shared > other active > archived)
4. If total results > 15 → set `was_truncated: true`, return only first 15
5. Return sorted list. Skill uses `was_truncated` to show message: "Found {total_count} entries, showing first 15. Refine your query."

#### Tool: `search_author`
**Spec compliance:** F4 (filter by author)

```
Input: {
  author_query: string,       // author name from query (e.g., "Vika")
  project?: string            // if specified — search only in this project
}
Output: {
  results: [
    { project: string, file: string, title: string, author: string, date: string }
  ],
  total_count: number,
  cached: boolean,             // whether cache was used
  warning: string              // ALWAYS: "Searching by author requires reading files and may take a few seconds"
}
```

Logic:
1. Check author index cache (by project)
2. If cache empty → read metadata of all entries (lines `- **Author:**`) from active project + _shared
3. Compare author_query with Author field — case-insensitive, substring match
4. Update cache
5. Return results

#### Tool: `search_deep`
**Spec compliance:** F4 (deep search)

```
Input: {
  query: string               // full-text search query
}
Output: {
  results: [
    { project: string, file: string, match_fragment: string }
  ],
  total_count: number,
  warning: string              // always: warning about indexing delay
}
```

Logic:
1. `GET /search/code?q={query}+repo:{owner}/{repo}+extension:md`
   **Rate limit:** GitHub Search API has a stricter limit of 10 requests/minute
   (vs 5000/hour for REST API). On 403 with `retry-after` → return `rate_limit`
   error with `retry_after_ms` from header. Do NOT use plugin-throttling for this
   endpoint — handle manually in github-client.js.
2. Parse results, exclude root.md and _meta.md
3. Add warning about indexing delay
4. Return with match fragments

#### Tool: `list_projects`
**Spec compliance:** F6

```
Input:  {}
Output: {
  projects: [
    { name: string, entries_count: number }
  ],
  archived: string[],
  active_project: string | null  // from state file
}
```

#### Tool: `switch_project`
**Spec compliance:** F6

```
Input:  { project: string }
Output: {
  status: "switched" | "created" | "not_found",
  project: string,
  entries_count: number,          // REQUIRED: number of entries in root.md
  last_entry_date: string | null, // REQUIRED: date of last entry or null if empty
  summary: string,                // REQUIRED: ready string "Project X: N entries, last — YYYY-MM-DD"
  root_content?: object           // content of root.md (if switched/created)
}
```

Logic:
1. Slugify entered name
2. Check folder existence
3. If exists → read root.md, update state file
   - `entries_count` = number of data rows in root.md table (excluding header and separator)
   - `last_entry_date` = parse Description column of last row for date pattern `(YYYY-MM-DD)`;
     if no date found in Description → use git commit date of last table row's file via `getLastCommitForFile()`; if that also fails → null
   - `summary` = conditional format:
     - If entries_count=0: "Project {name}: empty for now. Create the first entry to get started"
     - If entries_count>0 AND last_entry_date exists: "Project {name}: {entries_count} entries, last — {last_entry_date}"
     - If entries_count>0 AND last_entry_date=null: "Project {name}: {entries_count} entries"
4. If not → return `not_found` (skill asks user if should create)
5. If user confirms creation → tool creates folder + root.md from _meta.md template in a single atomic commit → return `status: "created"` with empty entries_count=0

**Note:** `switch_project` handles BOTH switching and creation (after user confirmation via skill). The "created" status is returned only when a new project folder is initialized.

**Race condition guard:** Before creating a new project, re-verify it still doesn't exist (another user may have just created it). If it now exists → skip creation, switch to it instead. Template population: replace `{project_name}` in _meta.md template with the slugified project name.

**Context loss recovery:** After context loss, if state file has active_project, show brief confirmation:
RU: "Память восстановлена. Работаю с проектом [name]."
EN: "Memory recovered. Working with project [name]."

#### Tool: `get_state`
**Spec compliance:** F8

```
Input:  {}
Output: { active_project: string | null, version: number }
```

#### Tool: `check_duplicate`
**Spec compliance:** F3 (duplicate check)

```
Input: {
  project: string,
  title: string,
  tags: string[],
  description: string
}
Output: {
  has_duplicate: boolean,
  candidates: [
    {
      file: string,
      name: string,
      description: string,       // description from root.md (so skill can show context)
      common_tags: string[],     // which tags matched
      keyword_overlap: number,   // 0-100, percentage of keyword overlap
      match_reason: string       // human-readable: "2 common tags: [auth, decision]" or "75% keyword overlap"
    }
  ]
}
```

Logic:
1. Read project root.md
2. For each entry:
   a. Tag matching: exact match (tag "auth" ≠ "auth-jwt"). Count common tags.
   b. Keyword overlap: extract words from input `title` + `description` (split by spaces, lowercase, remove stopwords). Count how many appear as substrings (case-insensitive) in existing entry's Description. `keyword_overlap = (matched / total_input_keywords) * 100`.
3. If ≥2 common tags OR keyword_overlap ≥50% → duplicate candidate
4. Format `match_reason`: "2 common tags: [auth, decision]" or "75% keyword overlap"
5. If no candidates → return `{ has_duplicate: false, candidates: [] }` (Skill can say "No similar entries found")

**Limitation:** check_duplicate is a recommendation check, not a guarantee of uniqueness. Between calling check_duplicate and write_entry, another user could create a similar entry. This is an acceptable compromise — full atomicity of check+write would require GitHub-level locking, which is unjustifiably complex.

## 4. Internal MCP Server Modules

### 4.1 github-client.js

Wrapper over Octokit with retry logic and error handling.

```javascript
// Initialization (current for @octokit/rest v22 + plugin-retry v8 + plugin-throttling v11):
import { Octokit } from "@octokit/rest";
import { retry } from "@octokit/plugin-retry";
import { throttling } from "@octokit/plugin-throttling";

const MyOctokit = Octokit.plugin(retry, throttling);

const octokit = new MyOctokit({
  auth: process.env.GITHUB_TOKEN,
  throttle: {
    onRateLimit: (retryAfter, options, octokit, retryCount) => {
      octokit.log.warn(`Rate limit hit for ${options.method} ${options.url}`);
      if (retryCount < 1) return true; // retry once
    },
    onSecondaryRateLimit: (retryAfter, options, octokit) => {
      octokit.log.warn(`Secondary rate limit for ${options.method} ${options.url}`);
      // don't retry — too aggressive
    },
  },
  retry: { doNotRetry: ["429"] }, // 429 handled by throttling plugin
  request: { timeout: 10000 },
});
```

```
Methods:
- getFileContent(path) → { content, sha }
- getDirectoryListing(path) → string[]
- getUserInfo() → { name, login }
- searchCode(query) → results[]
- getHeadSHA() → string
- getLastCommitForFile(path) → { author, date }
```

### 4.2 root-parser.js

Parsing and updating markdown tables in root.md. **This is the most fragile module — requires ≥15 unit tests.**

```
Methods:
- parseRootMd(markdown) → { description, entries[] }
- addEntryToRoot(markdown, entry) → updated_markdown
- updateEntryInRoot(markdown, filename, changes) → updated_markdown
- escapeTableCell(text) → escaped_text    // replace | with \|
- unescapeTableCell(text) → text          // reverse replace \| with |
- splitTableRow(line) → string[]          // split row respecting escaped pipes

  Escaped-pipe-aware row splitting algorithm:
  1. Remove leading/trailing `|` from the line
  2. Walk character by character:
     - If char is `\` and next char is `|` → append `|` to current cell, skip next char
     - If char is `|` → push current cell (trimmed) to result, start new cell
     - Otherwise → append char to current cell
  3. Push final cell (trimmed) to result
  4. Return array of cell values

Entry format:
{ name, file, description, tags[] }

CRITICAL REQUIREMENTS FOR PARSER:
1. Determine column order from header (Entry, Description, Tags can be in any order)
2. Correctly handle escaped \| inside cells
3. Skip separator line (---|---|---) in any format (with/without spaces)
4. addEntryToRoot() MUST be idempotent: if row with this filename in Entry column exists → don't add again (protection on retry after SHA conflict). Check by filename, not content.
   - Returns `{ updated_markdown, was_added: boolean }` — `was_added=false` means row already existed (idempotent skip)
   - If filename exists but description/tags differ → skip silently (do NOT update existing row — that's update_entry's job)
   - Skill uses `was_added` to detect retry scenarios without additional logic
5. Correctly parse markdown links [name](file.md) in Entry column
6. Handle empty lines and trailing whitespace

Edge cases for unit tests:
- Empty table (only header and separator)
- Table with 1 entry
- Cell with escaped | inside
- Description with special characters (quotes, parentheses)
- Tags with hyphens (react-query, e2e-testing)
- Invalid table (missing column)
- Unicode in description
```

### 4.3 slugify.js

```
Methods:
- slugify(title) → string
  Rules:
  1. Transliteration of non-Latin characters (via transliteration): "Привет" → "privet"
  2. Lowercase
  3. Replace everything except a-z, 0-9 with hyphens
  4. Remove leading/trailing hyphens and duplicate hyphens (-- → -)
  5. Max 60 characters
  6. Check reserved: root, _meta, _shared, shared → add suffix "-entry"

- ensureUnique(slug, existingFiles) → string
  If slug.md exists → slug-2.md, slug-3.md...
```

### 4.4 atomic-commit.js

Git Trees API workflow.

```
Methods:
- atomicCommit({ files[], message, parentSHA }) → { commitSHA, success }

  Steps:
  1. GET /repos/{owner}/{repo}/git/commits/{parentSHA} → get treeSHA of parent commit
  2. For each file: POST /repos/{owner}/{repo}/git/blobs → blobSHA
  3. POST /repos/{owner}/{repo}/git/trees → treeSHA
     IMPORTANT: pass base_tree = treeSHA from step 1 (to preserve all existing files!)
     tree: [{ path, mode: '100644', sha: blobSHA, type: 'blob' }, ...]
     NOTE: `path` is the FULL repo-relative path (e.g., "project-alpha/rive-vs-lottie.md",
     not just "rive-vs-lottie.md"). The Trees API handles nested directories automatically —
     no need to create intermediate tree objects for subdirectories.
  4. POST /repos/{owner}/{repo}/git/commits (tree: treeSHA, parents: [parentSHA]) → commitSHA
  5. PATCH /repos/{owner}/{repo}/git/refs/heads/main (sha: commitSHA, force: false)

  If step 5 returns 422 (SHA mismatch) → throw ConflictError

  Retry behavior: on retry, use fresh HEAD SHA and base_tree from new parent commit.
  This ensures the retried commit includes any concurrent changes from other users.
  The addEntryToRoot idempotency check prevents duplicate rows if previous attempt
  partially succeeded. After 3 failed retries, return status='conflict' to skill.
  Skill shows error and offers manual retry — user's entry data is NOT lost
  (it's still in the original request parameters).

  **write_entry vs update_entry on conflict:**
  - write_entry: automatic blind retry (new entries — low risk of overwriting)
  - update_entry: explicit user confirmation via `concurrent_edit` status (modifications — higher risk)

  Message format: "[shared-memory] {action}: {description}"
  Allowed actions: init, create-entry, update-entry, create-project

Retry strategy (for all MCP tools):

  HTTP Code   | Retry?               | Backoff              | Max Attempts
  ------------|----------------------|----------------------|-------------
  422 (SHA)   | YES (atomic-commit)  | exponential (1s,3s,9s)| 3
  429 (rate)  | YES (throttle plugin)| wait Retry-After     | 1
  5xx         | YES (retry plugin)   | exponential (1s,3s,9s)| 3
  timeout     | YES (retry plugin)   | exponential (1s,3s,9s)| 3
  4xx (other) | NO                   | —                    | —

  Retry responsibility division:
  - plugin-retry: 5xx, timeout (config: doNotRetry: ["429"])
  - plugin-throttling: 429 (callback onRateLimit, max 1 retry)
  - atomic-commit.js: 422 SHA mismatch (own retry-loop, up to 3 attempts)

Throttling: maximum 5 parallel requests to GitHub API (via p-limit v7+).
  Note: in p-limit v7 `activeCount` now counts actually executing promises (not queued). `pendingCount` — waiting in queue.
```

### 4.5 state-manager.js

Session state file management.

```
File: {workdir}/.shared-memory-state

Methods:
- readState() → { active_project: string | null, version: 1, author_cache?: {} }
- writeState(state) → void
  Atomically: write temp → rename

- getAuthorCache(project) → Map<filename, author> | null
- setAuthorCache(project, cache) → void
- invalidateAuthorCache(project?) → void   // if project not specified — clear entire cache

Error handling:
- File doesn't exist → return { active_project: null, version: 1 }
- File corrupted (invalid JSON) → log warning, delete and recreate with defaults
- active_project points to non-existent folder → set active_project to null, update file
- version field missing or not a number → treat as version 1, rewrite with version field

author_cache lifecycle:
- Initialization: cache empty on session start (lazy — filled on first search_author call)
- Invalidation: on switch_project (F6), write_entry (F3), update_entry (F5)
- New projects: start with empty cache naturally (lazy init — no special handling needed)
- No TTL: cache is session-only, no expiration within session. Stale entries from concurrent users are acceptable (session-level consistency, not real-time)
- Scope: per-project (mobile-app cache not cleared on write to backend-api)
- On project deletion detection (state points to non-existent folder): invalidate that project's cache
```

# Claude Shared Memory

You have access to a shared team knowledge base stored in a GitHub repository.
Use the shared-memory MCP tools (prefixed as `mcp__shared-memory__*` in your
tool list; internally registered as `read_root`, `write_entry`, etc.).

## Core Principles

1. **Never write automatically.** Only write to shared memory when the user
   explicitly asks. Triggers (RU): "запомни для команды", "добавь в общую память",
   "сохрани решение". Triggers (EN): "save for the team", "remember this for everyone".
2. **Always re-read.** Never rely on previously loaded data. Call `read_root`
   before every search or write operation.
3. **Explicit source attribution.** When answering from shared memory, say:
   - RU: "Согласно общей памяти команды, ..."
   - EN: "According to the team's shared memory, ..."
4. **Separation from local memory.** Shared memory ≠ your built-in memory.
   If the user says "запомни" / "remember this" without clarification, ask:
   - RU: "В общую память команды или в локальную?"
   - EN: "Should I save this to the team's shared memory or just my local memory?"

## Workflow: First-Time Onboarding

On the very first session (no state file exists yet AND `connect_repo` returns `status: "initialized"`):

1. Greet the user briefly:
   RU: "Общая память команды подключена! Репозиторий инициализирован."
   EN: "Team shared memory connected! Repository initialized."
2. Explain in 2-3 sentences what they can do:
   RU: "Используйте /remember чтобы сохранить решение, /memory чтобы найти что-то, /project чтобы переключить проект."
   EN: "Use /remember to save a decision, /memory to find something, /project to switch projects."
3. Suggest next steps with concrete dialogue:
   RU: "Хотите создать первый проект? (например, 'mobile-app')
   Или сохранить знание в общую память (_shared — доступно всем)?"
   EN: "Would you like to create a new project? (e.g., 'mobile-app')
   Or save knowledge to shared memory (_shared — available to everyone)?"

This only fires once per repository (detected by `connect_repo` returning `status: "initialized"`). Subsequent sessions skip to the normal Startup & Connection workflow.

## Workflow: Startup & Connection

On first invocation in a session:

1. Call `get_state` — check if there's an active project from previous session
2. Call `connect_repo` — connect to the GitHub repository
3. If `connect_repo` returns `status: "error"`:
   - Auth error → tell user: "Access token is invalid. Please update it in plugin settings."
     / "Токен доступа невалиден. Обновите в настройках плагина."
   - Repo error → tell user: "Repository not found. Check URL and token."
     / "Репозиторий не найден. Проверьте URL и токен."
   - **Continue without memory** (fallback mode) — do not block the user's workflow.
4. On success → read `_shared/root.md` and active project from state

## Workflow: Reading Context

When the user asks a question that might be in shared memory:

1. Call `get_state` to check active project
2. Call `read_root` for the active project (if any) and for `_shared`
3. Extract keywords from the user's question (use your judgment, not mechanical split)
4. Match keywords against Tags (exact) and Description (substring) — see references/matching-algorithm.md
5. Load relevant entries with `read_entry`:
   - 1 candidate → load automatically
   - 2-5 candidates → load all (context budget: max 5 entries at once).
     Show progress: "Loading N relevant entries from memory..." / "Загружаю N записей из памяти..."
   - >5 candidates → show list with descriptions and project names, ask user to select (max 5)
   **Hard limit:** Never load more than 5 entries in a single operation. If skill needs more context, ask user to narrow the query or select specific entries.
6. Answer the user, citing the source. Always show project name next to each entry:
   "**[title]** (project-name)" or "**[title]** (_shared)"

## Workflow: Writing

When the user asks to save something:

0. **Check intent — local vs shared:**
   - Explicit shared trigger ("remember for the team", "запомни для команды") → proceed to step 1
   - Explicit local trigger ("remember just for me", "запомни для меня") → use local memory, skip shared workflow
   - Ambiguous ("remember this", "запомни") → ask once:
     RU: "В общую память команды или в локальную?"
     EN: "Save to the team's shared memory or just my local memory?"
1. **Determine target project:**
   - If user explicitly names the project ("в проекте Alpha" / "in project Alpha") → use it
   - If `active_project` exists in state → ask:
     RU: "Сохранить в [project] или в другой проект?"
     EN: "Save to [project] or a different project?"
   - If no active project → show combined single prompt (avoid sequential questions):
     RU: "Где сохранить? В общую память (_shared — для всех) или в проект? [список проектов]"
     EN: "Where should I save this? Shared memory (_shared — for everyone) or a project? [project list]"
     If project selected → call `switch_project` first
2. Call `check_duplicate` to look for existing entries
3. If duplicate found → show WHY it matched and ask user:
   RU: "Похожая запись уже существует: **[name]** (совпадение: теги [common_tags], [keyword_overlap]% совпадение ключевых слов). Создать новую или обновить существующую?"
   EN: "A similar entry already exists: **[name]** (match: tags [common_tags], [keyword_overlap]% keyword overlap). Create a new one or update the existing one?"
   This helps the user make an informed decision without needing to read the existing entry first.
4. Prepare: title, tags (suggest from existing — see tag suggestions below), description (max 80 chars), content
5. Call `write_entry` with `auto_related: true`
6. If output contains `related_candidates` (>3 matches) → show list to user, ask to choose, then retry with `related_override`
7. Confirm to user with entry name and project

**UX optimization — minimize sequential questions:**
When the skill has enough context to infer the target project (active project exists,
or user explicitly said "shared"), skip the project selection question. Prefer
reasonable defaults over extra confirmation steps. The goal: most writes should
require at most 1 confirmation (the final "save?" prompt), not 3-4 sequential questions.

## Workflow: Updating

When user says "обнови" / "update":

1. Find the entry via `search_tags` or `read_root`
2. If multiple candidates → show list, ask to choose
3. Call `read_entry` to get current version — **save the returned `sha`**
4. Apply changes to the content
5. Call `update_entry` with `previous_sha` = SHA from step 3
6. If response `status: "concurrent_edit"` → show user `diff_summary` and clarify consequence:
   RU: "Эта запись обновлена [author] [date]. Что изменилось: [diff_summary].
   Если продолжите, ваши изменения заменят ВЕСЬ текст записи. Продолжить?"
   EN: "This entry was updated by [author] on [date]. What changed: [diff_summary].
   If you proceed, your changes will replace the ENTIRE entry text. Continue?"
   If user says "no" → offer options:
   RU: "Хорошо. Могу: (1) Загрузить текущую версию для просмотра, (2) Отменить — ваш текст сохранён в чате."
   EN: "OK. I can: (1) Load the current version for review, (2) Cancel — your text is saved in our chat."
   If user wants to see full current version → call `read_entry` again with current_sha.

## Workflow: Searching

See references/matching-algorithm.md for the full algorithm.

**Progress indicators:** For operations expected to take >1 second (connect_repo, search_deep, search_author on cache miss), show:
RU: "Ищу... (это может занять несколько секунд)"
EN: "Searching... (this may take a few seconds)"

**Search mode decision tree:**
1. Query contains a person's name ("что делала Вика?" / "what did Vika do?") → `search_author`
2. Query contains specific technical terms, tags, or topics → `search_tags` (fast)
3. Fast search returned no results → offer `search_deep` (full-text)
4. User explicitly asks for deep search ("поищи глубже" / "full text search") → `search_deep`

**Presenting results (spec F4):**
- ≤5 entries → show all with brief description from root.md
- 6–15 entries → compact list (title + project)
- >15 entries → first 15 + message:
  RU: "Найдено N записей, уточните запрос"
  EN: "Found N entries, please refine your query"

**Tag suggestions when writing (spec F3):**
When creating an entry:
1. First, show the 5 most-used tags in the active project:

   EN: "Common tags in this project: auth, decision, performance, stack, migration. Which apply?"
2. If user proposes a new tag similar to existing, suggest the existing one:
   RU: "Новый тег 'authentication'. Похожий существующий: `auth`. Использовать его?"
   EN: "New tag 'authentication'. Similar existing: `auth`. Use it instead?"

- On switch → call `switch_project`, show summary IMMEDIATELY (before next user question):

  RU: "Проект mobile-app: пока пусто. Создайте первую запись!"

- If project doesn't exist → ask to create

### Out-of-Scope Operations

The following operations are intentionally **not implemented** (per spec):

- **Delete entries:** Deletion is manual-only, by a repository admin through GitHub UI. The plugin never deletes files or rows from root.md.
- **Bulk operations:** Creating/updating multiple entries at once. Each entry is created individually through the skill workflow.
- **Semantic deduplication:** Detecting duplicates by meaning (e.g., "animation-guidelines" vs "micro-interactions"). Only tag-based and keyword-based duplicate detection is supported via `check_duplicate`.
